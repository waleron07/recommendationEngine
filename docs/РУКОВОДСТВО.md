# recoengine — руководство по подключению и использованию

Практическое, на русском. Как присоединить библиотеку к проекту, что подать ей на вход,
как получить выдачу и как она устроена внутри. Полное обоснование архитектурных решений —
в [ARCHITECTURE.md](../ARCHITECTURE.md).

> **Это не ИИ и не LLM.** Это детерминированная ранжирующая машина: даёшь пользователя,
> историю и набор кандидатов — получаешь упорядоченный список, где каждый объект умеет
> объяснить, почему он там, — данными, а не текстом.

---

## 1. Идея за одну минуту

Библиотека повторяет разделение, на котором стоит Elasticsearch: **маппинг знает домен,
скоринг знает математику, и они не пересекаются.**

Доменное знание превращается в числа внутри **экстракторов фич** (`FeatureExtractor`).
Всё, что ниже по конвейеру — скоринг, нормализация, ранжирование, диверсификация,
объяснение — работает только с числами. Стратегия скоринга физически не может дотянуться
до `item.payload.artist`: ей никогда не передают payload, и это проверяет компилятор.

Два следствия — ради них всё и затевалось:

- **Один движок обслуживает музыку, e-commerce и новости.** Меняешь экстракторы —
  остальное не трогаешь.
- **Ошибки всплывают на старте, а не в 3 часа ночи.** Стратегия, которой нужна фича,
  которую никто не поставляет, падает на `build()`, а не на запросе.

## 2. Как подключить

### Сейчас (pre-alpha, в npm ещё не опубликовано)

Пакеты пока не в реестре. Внутри этого монорепозитория они соединены через
workspace-протокол (`"@recoengine/core": "workspace:*"`), а во внешний проект их можно
взять из исходников:

```bash
git clone https://github.com/waleron07/recommendationEngine.git
cd recommendationEngine
pnpm install
pnpm verify        # линт + проверка архитектуры + сборка + тесты
pnpm build         # соберёт dist/ во всех пакетах
```

Дальше — либо импортировать из `packages/*/src` (в TS-проекте с тем же тулчейном), либо
из собранного `packages/*/dist`. Для локальной интеграции без публикации удобно
`pnpm link` или зависимость по пути:

```jsonc
// package.json потребителя
{
  "dependencies": {
    "@recoengine/core": "file:../recommendationEngine/packages/core"
  }
}
```

### Потом (после публикации в npm)

Появится публикация в npm, и раздел станет таким:

```bash
npm i recoengine                 # «с батарейками»: фасад, реэкспортит ядро
# или точечно:
npm i @recoengine/core @recoengine/strategies @recoengine/modifiers
```

`@recoengine/core` — ядро с **нулём зависимостей**. `recoengine` — фасад для быстрого
старта. Остальные пакеты (`strategies`, `modifiers`, `diversity`, `features`, `testing`) —
готовые наборы, которые подключаются по необходимости.

### Требования к среде

- **Node ≥ 20** (работает также на Bun и Deno — CI это проверяет).
- **ESM.** Все пакеты `"type": "module"`; импорты — с расширением `.js`.
- Ядро изоморфно: не использует ни `process`, ни `Buffer`, ни `window` — можно тащить и
  в браузер.

## 3. Какие данные нужны

Движку нужны три вещи на запрос — **пользователь, история, кандидаты** — плюс, один раз
при сборке, описание того, как превращать домен в числа.

### 3.1 Кандидаты — `CandidateProvider`

Единственное место, которому разрешено трогать вашу базу. Отдаёт «сырые» объекты `Item`:

```ts
interface Item<P> {
  readonly id: ItemId            // itemId('track:123')
  readonly type: string          // 'track' | 'product' | 'article' — свободная строка
  readonly payload: P            // ваши доменные данные; движок внутрь НЕ смотрит
}
```

Бюджет (`budget.maxItems`, `budget.deadline`) **проталкивается внутрь** провайдера — его
нужно превратить в `LIMIT` запроса, а не обрезать миллион строк после `SELECT`.

### 3.2 Пользователь — `User`

```ts
interface User<P> {
  readonly id: UserId            // userId('u1')
  readonly payload: P            // доменные данные о пользователе, опять же непрозрачны
}
```

### 3.3 История — `History` из событий `Event`

```ts
interface Event {
  readonly id: EventId
  readonly userId: UserId
  readonly itemId: ItemId
  readonly type: string          // 'play' | 'like' | 'purchase' | 'skip' — ваша семантика
  readonly at: Timestamp         // timestamp(Date.now()) — миллисекунды или ваша шкала
  readonly value?: number        // рейтинг, длительность, сумма заказа — по желанию
}

interface History {
  readonly userId: UserId
  readonly events: readonly Event[]
}
```

История индексируется один раз за запрос в `HistoryIndex` (`countFor`, `lastAtFor`,
`hasSeen`, `aggregate`) — стратегиям и модификаторам не нужно фильтровать массив событий
руками. Пустая история — это законный **cold start** (см. §7).

### 3.4 Фичи — что домен превращает в числа (`FeatureExtractor`)

Экстрактор объявляет, какие фичи он производит (`provides`), и заполняет их колонки,
читая payload. Это единственный компонент, который знает, что такое «трек»:

```ts
const popularity: FeatureExtractor<Track> = {
  id: 'popularity-extractor',
  version: '1.0.0',
  provides: [{
    key: featureKey('popularity_global'),
    kind: 'numeric',
    defaultValue: 0,                 // что значит отсутствие фичи — решает её владелец
    description: 'счётчик прослушиваний',
    owner: 'popularity-extractor',   // ОБЯЗАН совпадать с id экстрактора
    ownerVersion: '1.0.0',
  }],
  // Батчево по всему набору: один запрос, а не 5000.
  extract: async (set, out) => {
    const column = out.columnMut(featureKey('popularity_global'))
    for (let row = 0; row < set.size; row++) column[row] = set.at(row).item.payload.plays
  },
}
```

Готовые стратегии ожидают фичи с конкретными именами (`popularity_global`,
`affinity_*`, `item_age`, `sim_to_recent`, `cooc_score`, `item_familiarity`, …) — их
поставляет ваш экстрактор. Список — в §4.2 и в [ARCHITECTURE.md §11.3](../ARCHITECTURE.md).

## 4. Как пользоваться

### 4.1 Минимальный движок (пишем стратегию сами)

Этот пример проверяется тестом — [readme-example.test.ts](../packages/core/src/readme-example.test.ts),
он компилируется и выдаёт ровно показанное. Ваше здесь — только `db`.

```ts
import {
  createEngine, featureKey, itemId, rank, strategyId, userId,
  type CandidateProvider, type FeatureExtractor, type PreFilter, type ScoringStrategy,
} from '@recoengine/core'

interface Track { readonly title: string; readonly plays: number }
const POPULARITY = featureKey('popularity')

// 1. ОТКУДА КАНДИДАТЫ — единственный доступ к базе.
const library: CandidateProvider<Track> = {
  id: 'library', version: '1.0.0',
  provide: async (_ctx, budget) => {
    const rows = await db.tracks.findMany({ take: budget.maxItems })
    return rows.map((r) => ({ id: itemId(r.id), type: 'track', payload: { title: r.title, plays: r.plays } }))
  },
}

// 2. ЖЁСТКИЕ ПРАВИЛА до всего дорогого. approve() синхронна и тотальна; бросок = «нет».
const notPlayedYet: PreFilter<Track> = {
  id: 'not-played-yet', failClosed: true,
  approve: (candidate, ctx) => !ctx.history.hasSeen(candidate.item.id),
}

// 3. ДОМЕН → ЧИСЛА.
const popularity: FeatureExtractor<Track> = {
  id: 'popularity-extractor', version: '1.0.0',
  provides: [{ key: POPULARITY, kind: 'numeric', defaultValue: 0,
    description: 'прослушивания', owner: 'popularity-extractor', ownerVersion: '1.0.0' }],
  extract: async (set, out) => {
    const column = out.columnMut(POPULARITY)
    for (let row = 0; row < set.size; row++) column[row] = set.at(row).item.payload.plays
  },
}

// 4. МАТЕМАТИКА. О треках не знает — читает колонку чисел.
const popular: ScoringStrategy = {
  id: strategyId('popularity'), requires: [POPULARITY],
  normalizer: rank,   // стратегия знает свою шкалу: rank устойчив к выбросам
  score: (view) => ({ strategyId: strategyId('popularity'), raw: view.items.column(POPULARITY), reasons: new Map() }),
}

const engine = createEngine<Track>()
  .use(library).use(notPlayedYet).use(popularity).use(popular)
  .configure({
    limits: { maxCandidates: 5_000, maxLimit: 100, timeoutMs: 200 },  // обязательно, без дефолтов
    weights: { popularity: 1.0 },
  })
  .build()   // ← здесь упадёт, если фичи не хватает или конфиг не сходится

const result = await engine.recommend({
  user: { id: userId('u1'), payload: {} },
  history: { userId: userId('u1'), events: [] },
  limit: 10,
  explain: 'reasons',
})
```

### 4.2 С готовыми пакетами (не пишем стратегии руками)

Стратегии и модификаторы есть в виде фабрик — их не нужно писать
самому, достаточно поставлять нужные фичи. Тот же движок, но короче:

```ts
import { createEngine } from '@recoengine/core'
import { popularityStrategy, recencyStrategy } from '@recoengine/strategies'
import { fatigueModifier } from '@recoengine/modifiers'

const engine = createEngine<Track>()
  .use(library)                                   // ваш провайдер
  .use(popularityFeatures)                        // экстрактор, дающий popularity_global и item_age
  .use(popularityStrategy({ cohortFeature: null })) // требует popularity_global
  .use(recencyStrategy({ halfLife: 30 }))         // требует item_age
  .use(fatigueModifier({ threshold: 10 }))        // приглушает «заслушанное», читая историю
  .configure({
    limits: { maxCandidates: 5_000, maxLimit: 100, timeoutMs: 200 },
    weights: { popularity: 0.7, recency: 0.3 },   // ключ = id стратегии
  })
  .build()
```

Девять стандартных стратегий (`historyStrategy`, `affinityStrategy`, `popularityStrategy`,
`recencyStrategy`, `similarityStrategy`, `coOccurrenceStrategy`, `noveltyStrategy`,
`discoveryStrategy`, `contextStrategy`) и три модификатора (`fatigueModifier`,
`noveltyModifier`, `boostModifier`) — каждый доменно-нейтрален и параметризуется, какую
фичу читать. Разбор — в [ARCHITECTURE.md §11.3 и §15](../ARCHITECTURE.md).

### 4.3 Что на выходе

```ts
for (const { rank, item, score, explanation } of result.recommendations) {
  console.log(`${rank}. ${item.payload.title} — ${score.toFixed(1)}`)
  for (const c of explanation.contributions) {
    console.log(`     ${c.strategyId}: raw ${c.raw} → ${c.contribution.toFixed(3)}`)
  }
}

// Диагностика — часть ответа, а не строка лога: даже пустая выдача объясняет себя.
console.log(result.diagnostics)
// { totalMs, retrieved, filtered, stages: [...16 таймингов], warnings: [] }
```

Каждая рекомендация несёт `rank`, `item`, `score` (0..100) и `explanation` с
`contributions` (кто и сколько добавил) и `reasons` (стабильные коды вроде
`favorite_artist` с параметрами — рендерятся вне ядра, чтобы i18n не жил в библиотеке).

## 5. Как это работает внутри

Один фиксированный конвейер из 16 стадий. Порядок не настраивается — расширяются
**порты**, а не последовательность (в конвейер, куда каждый может вставить фазу,
невозможно рассуждать «когда сработает мой фильтр»).

```
0.  request        сборка контекста: сигнал отмены, дедлайн, seed RNG
1.  retrieval      CandidateProvider'ы → набор кандидатов (с бюджетом)
2.  prefilter      PreFilter'ы: жёсткие правила по payload, fail-closed
3.  extraction     FeatureExtractor'ы: домен → колонки чисел (+ ProfileVector о юзере)
4.  engineering    FeatureTransform'ы: производные фичи
4b. postfilter     PostFilter'ы: правила, которым нужны фичи
5.  scoring        ScoringStrategy'ы: колонка сырых чисел на стратегию
6.  normalization  ScoreNormalizer'ы: каждую колонку в [0..1] (иначе веса — ложь)
7.  combination    взвешенная сумма: base = Σ(weight·norm) / Σ weight
8.  modifiers      ScoreModifier'ы: fatigue/novelty/boost — «уместно ли показывать сейчас»
9.  ranking        top-K через min-heap, детерминированный tie-break
10. diversification Diversifier (MMR, квоты) — чтобы не 20 треков одного артиста
11. blending       Blender: квоты слотов (exploration/discovery)
13. explanation    Explainer: след → Explanation по уровню ctx.explain
```

Ключевые инварианты:

- **Нормализация обязательна.** `popularity` вернёт 4 200 000, `recency` — 0.87. Без
  приведения к [0..1] взвешенная сумма даст только популярность, и это самая незаметная
  ошибка самодельных движков.
- **Деление на `Σ weights` держит веса честными.** Добавили девятую стратегию — масштаб
  не поехал; `artist: 0.9` против `popularity: 0.3` читается как «артист втрое важнее».
- **Объяснимость структурна.** Доска не умеет хранить score, не храня, из чего он собран,
  поэтому `Σ contributions = score` верно по построению, а не по дисциплине.
- **Детерминизм.** RNG сеется (xoshiro128**), тот же пользователь получает ту же выдачу —
  баг воспроизводится, A/B меряет вариант, а не планировщик.

## 6. Конфигурация

```ts
.configure({
  // Обязательно, без дефолтов: у «сколько моей базы может тронуть один запрос» нет
  // честного значения по умолчанию. Пропустишь — build() бросит INVALID_CONFIG.
  limits: { maxCandidates: 5_000, maxLimit: 100, timeoutMs: 200 },
  weights: { popularity: 0.7, recency: 0.3 },   // ключ = id стратегии; незарегистрированный ключ отвергается на build()
  errorPolicy: 'strict',                        // 'strict' в dev/CI, обычно 'degrade' в prod
})
```

Отмена: `ctx.signal` формируется всегда — из вашего сигнала и `timeoutMs`. Порт с I/O
обязан пробрасывать его вниз; порт с длинным CPU-циклом — звать `throwIfAborted()`.

## 7. Cold start и отказы

- **Cold start ничего не стоит.** Стратегия объявляет `applicable: (ctx) => ctx.history.size >= 20`,
  и новый пользователь просто получает те стратегии, что могут с ним говорить, а вес
  остальных перераспределяется. Никаких `if (isNewUser)` в ядре. Готовые
  `historyStrategy`/`affinityStrategy`/… уже так устроены; `popularity`/`recency` работают
  и на пустой истории — это и есть фолбэк.
- **Ничего не деградирует молча.** Упавший экстрактор по умолчанию валит запрос. Пометишь
  `criticality: 'optional'` — деградирует до `defaultValue` фичи, но со структурным
  warning в `diagnostics` и метрикой: незамеченная деградация — это медленный регресс
  качества.
- **Фильтры fail-closed по контракту.** Что не одобрено явно — не показывается. При этом
  единичный сбой удаляет кандидата, а не запрос; систематический (доля выше
  `filterErrorBudget`) — валит запрос громко.
